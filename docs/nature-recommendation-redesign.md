# 性格推荐重设计草案

> 本文是性格推荐规则的设计背景与调参记录，不等同于当前代码的完整实现规范。当前运行逻辑以 `src/domain/nature.js` 为准；本文用于解释规则演进、未决问题和样例讨论。


本文档是一版可讨论的性格推荐设计，用于后续多轮规则迭代。它先固定“页面应该解释什么、推荐系统应该输出什么、规则如何分层”，再进入 `src/domain/nature.js` 与 `src/components/nature.jsx` 的实现。

## 设计目标

1. **从“给一个最高分”改为“给完整判断”**：展示全部性格候选，不再只展示 top-N；每个候选都要能看到修正后的六维与推荐结论。
2. **从“简单强弱差”改为“定位驱动”**：先判断精灵定位，再评价某个性格是否强化了核心能力、是否牺牲了可接受的弱项。
3. **解释要能被讨论和调参**：每条结论都要拆成可阅读的推荐理由 / 不推荐理由，后续调整规则时能明确是哪一层规则在起作用。
4. **不改变数据模型**：本轮设计不需要 Dexie schema 变更，不新增持久化表；性格推荐仍是基于当前输入即时计算。
5. **保持本地优先和通用场景兼容**：洛克王国资料可以得到更好的默认体验，但普通资料表只要能识别六维，也应可使用基础推荐。

## 非目标

- 不接入进化链、属性克制、PVP 环境等对局向深度资料；技能只作为 d.json 的可查看资料与标签来源，不在本阶段实现完整对战模拟。`rocom-data` 的经验仅用于启发轻量技能效果字段与规则解释，预置资料仍以本仓库官方 `d.json` 同步结果为准。
- 不在这一版里做云端同步、AI 自动标注或用户共享规则库。
- 不把推荐结果写回资料库或收集记录；用户是否采纳由后续手动记录。

## 输入模型

### 必需输入

- 六维：生命、物攻、魔攻、物防、魔防、速度。
- 候选性格：按预置性格字段，枚举“强化一项、弱化另一项”的全部合法组合。
  - 生命参与强化/弱化。
  - 可修正维度为：生命、物攻、魔攻、物防、魔防、速度。
  - 总候选数：6 个增益维度 × 5 个减益维度 = 30 个合法修正性格；不允许同项增加/降低。

### 可选输入

- 特性标签：例如物攻倾向、魔攻倾向、速度倾向、防御、辅助、能量循环、异常控制等。
- 技能效果标签：由 `skillRefs` 指向的技能资料行提供，例如先手、速度、回复/吸血、护盾/减伤、能量、强化/削弱、异常控制、应对攻击/防御/状态、脱离轮转、连击、蓄力等。缺失时可按技能效果文本兜底解析。
- 资料行摘要：名称、编号、形态、系别、特性文本。
- 用户偏好（后续可加，不作为第一版实现前提）：偏向输出、偏向保守、偏向速度、偏向耐久、偏向均衡。
- 阵容上下文预留：阵容系统暂不实现，但推荐引擎可接收“阵容需要保留的性格 id/name”，把非硬性风险的候选至少保留为“可保留”。

## 输出模型

每个候选性格输出一个 `NatureEvaluation`：

```ts
type NatureDecision = 'recommended' | 'keepable' | 'notRecommended'

type NatureEvaluation = {
  nature: { raise: StatKey | null; lower: StatKey | null }
  name: string
  decision: NatureDecision
  score: number
  roleTags: string[]
  roleLabel: string
  adjustedStats: Record<StatKey, number>
  deltas: Record<StatKey, number>
  reason: string[]
  warning: string[]
}
```

页面展示时不需要暴露类型名，但应对应以下 UI 信息：

- 性格名称。
- 结论徽章：推荐 / 可保留 / 不推荐。
- 适合方向：例如物攻输出、魔攻输出、双攻输出、高速先手、耐久坦克、辅助控制、能量循环。
- 六维修正结果：原始值、修正后值、差值；强化项用正向颜色，弱化项用风险颜色。
- 推荐理由：为什么它适合当前定位。
- 不推荐理由 / 风险：它牺牲了什么，是否牺牲核心维度。

## 规则总览

推荐分为四层，从稳定到可调：

1. **基础数值分析**：只看六维分布，判断强项、弱项、速度档位和耐久水平。
2. **定位识别**：结合六维与特性标签，推断一个或多个适合方向。
3. **候选性格评价**：判断强化项是否命中核心能力，弱化项是否可承受。
4. **结论分档**：把分数和硬性风险转成推荐 / 可保留 / 不推荐。

这样做的好处是：后续讨论时可以单独改“定位识别”或“分档阈值”，不必重写整套 UI。

## 第 1 层：基础数值分析

对输入六维计算以下派生信息：

- `total`：六维总和。
- `attackBias`：物攻与魔攻差异。
- `defenseBias`：物防与魔防差异。
- `bulkScore`：生命、物防、魔防的综合耐久。
- `speedRank`：速度相对自身六维的位置。
- `topStats` / `bottomStats`：最高和最低的 1～2 个维度。

建议先使用相对规则，而不是固定阈值：

- 某项高于五维均值 10% 以上：视为强项。
- 某项低于五维均值 10% 以上：视为弱项。
- 物攻和魔攻差距低于 8%：可视为双攻倾向。
- 速度高于可修正五维均值 8%：高速倾向；低于 8%：低速倾向。

> 说明：第一版用相对阈值更适合不同数值规模。如果后续发现洛克王国有明确速度线或种族值档位，再补固定阈值。

## 第 2 层：定位识别

定位不是单选，而是一个有权重的标签集合。第一版建议输出以下方向：

| 定位 | 触发条件 | 关注的强化项 | 通常可弱化项 |
|---|---|---|---|
| 物攻输出 | 物攻明显高于魔攻，或特性标签偏物攻 | 物攻、速度 | 魔攻、较低防御 |
| 魔攻输出 | 魔攻明显高于物攻，或特性标签偏魔攻 | 魔攻、速度 | 物攻、较低防御 |
| 双攻输出 | 物攻和魔攻都不低，且差距小 | 速度、较高攻击项 | 较低防御，谨慎弱化任一攻击 |
| 高速先手 | 速度是强项，或特性标签偏速度 / 控制 / pivot | 速度、主攻 | 非核心攻击、防御短板 |
| 耐久坦克 | 生命 + 双防较高，攻击不突出 | 物防、魔防、主防项 | 非核心攻击、速度 |
| 辅助控制 | 特性标签偏辅助 / 控制 / 能量循环 | 速度、耐久 | 非核心攻击 |
| 能量循环 | 特性标签偏能量循环，速度或耐久不差 | 速度、耐久 | 非核心攻击 |
| 均衡泛用 | 无明显强弱倾向 | 均衡或强化轻微强项 | 避免牺牲最低短板 |

定位权重建议：

- 数值触发给 1.0～2.0 权重。
- 特性标签触发给 0.5～2.0 权重。
- 同时被数值和标签命中时，作为主要定位。
- 页面展示前 1～2 个最高定位即可，避免文案过载。

## 第 3 层：候选性格评价

每个候选性格按以下维度加减分：

### 加分项

- 强化主要定位的核心维度：大幅加分。
  - 物攻输出强化物攻。
  - 魔攻输出强化魔攻。
  - 高速先手强化速度。
  - 耐久坦克强化物防或魔防。
- 强化当前强项：中幅加分。
- 强化特性标签指向的维度：中幅加分。
- 弱化明确非核心、且已经偏低或不参与定位的维度：小到中幅加分。
- 若整体定位接近均衡，则更偏向强化轻微强项、弱化非核心项；不再单独增加“均衡性格”。

### 减分项

- 弱化主要定位的核心维度：大幅减分。
- 弱化当前最高项或特性强指向维度：中到大幅减分。
- 双攻输出弱化物攻或魔攻：中到大幅减分。
- 速度是主要定位时弱化速度：大幅减分。
- 耐久是主要定位时弱化明显短板防御：中幅减分。
- 强化已判定为非核心弃用维度：小到中幅减分。

### 硬性风险标记

以下情况即使总分不低，也要加 warning，并可能降档：

- `lower` 命中主要定位核心维度。
- `raise` 和主要定位完全无关，且 `lower` 不是可接受弱项。
- 双攻定位下弱化任一攻击，除非另一攻击明显更高且特性强烈单攻倾向。
- 当前没有有效六维时，不输出推荐，只显示空态。

## 第 4 层：结论分档

第一版建议分档逻辑：

- **推荐**：分数位于前 20%，且无硬性风险；通常 3～5 个。
- **可保留**：分数处于中间区间，或有轻微风险但仍符合某个定位；通常 8～12 个。
- **不推荐**：分数较低，或命中硬性风险；通常 5～10 个。

如果用户希望更明确，可以后续改成固定数量：

- 推荐：最多 5 个。
- 可保留：中间 8～10 个。
- 不推荐：其余。

但第一版更建议用阈值 + 风险，因为不同精灵的可接受候选数量可能不一样。

## UI 草图

### 顶部输入区

- 左侧：六维输入，保持现有手动输入。
- 右侧：洛克王国精灵资料带入，固定读取「精灵基础资料」；技能通过 `skillRefs` 关联「技能资料」后自动参与分析。
- 输入区下方：展示识别到的定位标签与一句摘要：
  - 示例：`主要定位：物攻输出 / 高速先手`
  - 示例：`理由：物攻与速度处于六维前列，特性标签也偏向先手输出。`

### 推荐摘要区

展示 3 块信息：

1. **首推性格**：评分最高且无硬性风险的推荐项。
2. **推荐结论分布**：推荐 N 个、可保留 N 个、不推荐 N 个。
3. **主要风险提醒**：例如“弱化速度会明显影响先手定位”。

### 全部候选区

候选区按结论分组，而不是只按分数一列到底：

1. 推荐。
2. 可保留。
3. 不推荐。

每个候选卡片显示：

- 性格名称。
- 结论徽章。
- 分数。
- 适合方向。
- 强化 / 弱化维度。
- 一句最关键理由。

点击候选后，在右侧或下方展开详情：

- 六维条形对比：原始 → 修正后。
- 推荐理由列表。
- 风险 / 不推荐理由列表。

## 文案规范

推荐理由避免只说“分数最高”，应使用可验证的句式：

- `强化物攻命中「物攻输出」核心维度。`
- `弱化魔攻的风险较低，因为当前定位主要依赖物攻。`
- `速度已是主要优势，继续强化可提高先手价值。`
- `弱化速度会削弱「高速先手」定位，因此降为不推荐。`
- `双攻差距较小，弱化任一攻击都会降低泛用性。`

## 本轮补充共识：速度线、技能库与规则引擎

### 速度线要考虑双方性格修正

速度线不能只看“所有精灵的基础速度”，因为实战里对手也可能加速或减速。第一版速度分析建议分两层：

1. **基础速度档位**：用全图鉴基础速度分位数判断该精灵天然处于极慢 / 偏慢 / 中速 / 偏快 / 高速 / 超高速。
2. **修正后速度档位**：对每个候选性格计算 `floor/round(base × 1.1)` 或 `base × 0.9` 后，再看它跨过了哪些速度分位线。

对“速度收益”的解释不要只写“速度 +10%”，而要写成：

- 是否跨过 P75 / P90 / P95 等关键档位。
- 减速后是否跌出当前档位。
- 如果未来有对战环境数据，再进一步比较“能超过哪些常见对手的加速 / 无修正 / 减速形态”。

因此速度线规则应同时生成三组参考值：

```text
本体无修正速度
本体加速后速度
本体减速后速度
```

并预留对手侧的三组速度：

```text
对手减速速度线 / 对手无修正速度线 / 对手加速速度线
```

只有当“加速后跨过关键对手线”或“减速后仍不丢关键线”时，速度修正才应被高评价；否则只按泛用速度档位给中等权重。

### 双攻高不必默认走双攻玩法

“双攻都高”只说明这个精灵有双攻潜力，不代表一定要推荐双攻玩法。规则上应拆成两种路线：

| 路线 | 判断方式 | 对减攻击项的态度 |
|---|---|---|
| 双攻玩法 | 技能池两侧都有关键输出，或特性/技能鼓励混合攻击 | 减任一攻击都谨慎 |
| 单攻玩法 | 一侧攻击更契合技能池、特性、能耗或打法 | 另一侧可以成为可牺牲项 |

也就是说，双攻高时不要直接把 `patk` / `matk` 都标成不可牺牲，而是先问：

1. 技能池是否真的需要两侧攻击？
2. 是否存在强力单攻路线？
3. 另一侧是否只是“数值高但没有关键技能”？

如果答案偏向单攻，那么另一侧攻击可以被视为可牺牲属性。

### 捕捉建议要偏保守，但必须有定位依据

单只精灵捕捉建议和最终极限培养不同：捕捉过程中多保留几个性格通常不会显著增加成本，因此 `可保留` 档应承担“有玩法依据、抓到了可以先留”的作用。后续规则校准按以下口径分档：

- `推荐`：强化核心定位，弱化项明确低风险，且技能/特性/外部定位能支持。
- `可保留`：存在合理玩法分支，或双攻中一侧技能略弱但仍有面板/技能可能；不应因为不是最优就直接打死。
- `不推荐`：明显伤害核心定位，例如高速输出减速度、耐久回复核心减生命、防御手减对应防御、明确单攻输出减主攻，或真双攻玩法成立时减任一攻击。

双攻精灵尤其要区分“稍弱”和“无关键技能”：一侧只是稍弱时，降低该侧攻击的性格优先进入 `可保留`；只有当技能组明显证明单攻路线时，才把降低另一攻视作低代价牺牲项。

### 公式辅助指标只用于校准定位线

rocom-data 的伤害公式提示输出更接近“攻击项 × 技能有效威力 ÷ 对手防御”，承伤更接近“生命 × 对应防御”，因此单纯用六维相加或比例会误判部分样例。当前不引入完整战斗模拟，但报告可增加轻量公式辅助指标：

- 物理有效输出线：`物攻 × 物理攻击技能平均威力`。
- 魔法有效输出线：`魔攻 × 魔法攻击技能平均威力`。
- 物理有效耐久：`生命 × 物防`。
- 魔法有效耐久：`生命 × 魔防`。
- 综合短板耐久：`生命 × min(物防, 魔防)`。

这些指标只作为“定位线 sanity check”，不代表真实伤害；真实对局仍会受属性克制、天气、应对、印记、能耗、技能组合和队伍环境影响。

当前规则允许公式辅助在双攻转单攻时提供轻量投票：当技能标签数量和公式辅助方向冲突时，不直接覆盖技能判断，但会降低削弱更高质量输出侧的性格分数，并允许削弱低质量输出侧的性格作为低优先可保留分支。

### 明确单攻路线下，牺牲弃用攻击项应形成硬支配

火神等高速物攻样例确认：当技能池、特性与公式辅助指标已经明确证明单攻路线时，非主路线攻击项（例如物攻路线下的魔攻）应视为优先牺牲项。同一强化维度内，如果存在“强化相同核心项 + 弱化弃用攻击项”的低风险候选，那么弱化生命、速度、物防、魔防或主攻的候选不应仅因分数尚可而保留；应被硬支配降入“不推荐”。

这条规则只在支配方本身无硬风险、且弱化项确实是当前技能路线的非主攻击项时触发；仍不把外部推荐性格直接抄入规则，也不对单只精灵做硬编码。

### 低速与短板防御的支配收紧

魔力猫等低速站场样例暴露出两个需要通用处理的问题：

- 基础速度低于 P25、主定位不是高速输出时，即使技能池有控制/先手线索，加速性格也只应获得低权重节奏收益，不能高于强化生命、主攻或防御的低代价组合。
- 站场型精灵降低当前短板防御时，必须比普通“降单防”更谨慎；若同强化项里存在降速度或降非主攻的低代价选择，降短板防御的候选更容易被支配。
- 如果低速非抢速精灵为了加速牺牲攻击、生命或防御核心项，应默认降到 `不推荐`，除非存在明确的高速/迅捷机制。
- “专项防御手”不能豁免降短板防御：强化一侧防御但削弱当前短板防御时，仍应优先被同强化项的低代价性格支配。

这两条规则仍服务捕捉保守口径：不是最优的性格可以进 `可保留`，但明显有更低代价替代、且削弱短板的组合不应长期留在高优先可保留。

### 防御强化可以服务“专项防御手”

防御不只用于“补短板”。有些精灵可能应强化本来就偏高的防御，把它做成专项防御手：

- 物防高 + 生命不低 + 技能/特性有护盾、减伤、反击、嘲讽、站场收益 → 可以强化物防，作为物理防御手。
- 魔防高 + 生命不低 + 技能/特性有回复、护盾、异常控制、站场收益 → 可以强化魔防，作为魔法防御手。
- 双防都高 + 生命高 → 生命强化也可能优于单防强化，因为生命同时放大两侧耐久。

因此防御评价要同时支持两种路线：

1. **补短板**：强化较低防御，改善综合耐久。
2. **做专精**：强化较高防御，明确承担单侧防御职责。

### d.json 技能接入方向

后续确认 `d.json` 会提供真实技能数据后，技能应进入系统，但不要直接把几十个技能平铺到主表格里。建议：

- 在精灵资料详情里增加“技能”折叠区，默认收起。
- 表格只显示技能摘要，例如：技能数、物理/魔法/变化数量、低能耗技能数、关键标签。
- 性格推荐工具读取技能摘要和技能标签，不要求用户逐条阅读全部技能。
- 原始技能列表保留可查看入口，方便用户核对推荐理由。

### 技能是否单独建资料库

建议把技能做成单独资料库，而不是只作为精灵行里的长字段：

- 技能是通用实体，同一个技能会被多个精灵共享。
- 单独资料库可以避免重复存储几十次相同技能文本。
- 精灵资料通过引用/关联保存“可学习技能列表”。
- 性格推荐读取时可把精灵关联到的技能展开成技能标签集合。

在 Dexie 层面，这仍可复用现有 `catalogTables` / `catalogFields` / `catalogRows`，不需要新建 IndexedDB 表：

```text
普通资料表：精灵基础资料
普通资料表：技能资料
关联字段：精灵 -> 技能列表 / 技能引用列表
```

UI 上技能资料表可以隐藏在普通资料库表选择里，或作为洛克王国场景的一个可切换资料表。具体是否默认展示，需要再讨论。

### 技能标签先从规则学习，不强依赖大模型

技能标签可以先通过规则和关键词学习，后续再人工修正。第一版建议关注这些标签：

| 标签 | 触发线索 | 对性格的影响 |
|---|---|---|
| `physicalMove` | 物理伤害 / 受物攻影响 | 支持物攻路线 |
| `magicalMove` | 魔法伤害 / 受魔攻影响 | 支持魔攻路线 |
| `statusMove` | 变化、控制、回复、护盾 | 支持辅助/耐久路线 |
| `lowCost` | 能耗低、能耗为 1、能量消耗少 | 提高速度/循环价值 |
| `highPower` | 高威力、威力提升、爆发 | 支持对应攻击项 |
| `priority` | 先手、优先、立即行动 | 降低速度依赖 |
| `slowBenefit` | 后手、后于对手、受到攻击后 | 降低减速度惩罚，甚至支持减速度 |
| `pivot` | 替换、返场、换入、换下 | 支持慢速中转或辅助定位 |
| `buffTransfer` | 传递、给予队友、场下获得 | 支持辅助/后手价值 |
| `shield` | 护盾、减伤、免疫、抵免 | 支持防御/生命强化 |
| `healing` | 回复、恢复、治疗、吸血 | 支持生命/耐久强化 |
| `control` | 冻结、中毒、灼烧、麻醉、恐惧、睡眠 | 支持速度或辅助定位 |
| `lowHpTrigger` | 低生命、濒死、保留 1 点生命 | 允许生命降低的例外 |
| `deathTrigger` | 退场、阵亡、牺牲、自爆 | 允许生命/防御降低的例外 |

### 大模型抽象能力如何写进工程

运行时推荐不应强依赖大模型，但工程可以给“大模型抽象能力”预留离线入口：

1. **离线标注辅助**：给脚本输入技能文本，让大模型建议技能标签。
2. **人工审核落盘**：建议标签必须写入可版本化的 JSON 或脚本规则，不直接在用户端实时调用模型。
3. **规则引擎只读标签**：运行时只读取已经确认的 `skillTags` / `traitTags`，保持本地优先、可复现、可测试。
4. **解释文案模板化**：大模型可以帮助设计模板，但实际文案由确定性模板生成。

这样既能利用大模型的抽象能力，又不会让推荐结果变成不可复现的黑盒。

## 样例规则与全性格分析：鸭吉吉（蓬松的样子）

为了让后续讨论有一个可落地的参照，先用「鸭吉吉（蓬松的样子）」跑一版自定义规则。该样例来自当前洛克王国预置资料：普通系、特性「挺起胸脯」，特性说明为「携带的能耗为1的技能，威力+50%。」，特性标签为 `patkLean`、`spdLean`、`defense`、`energyCycle`；六维为生命 136、物攻 95、魔攻 35、物防 55、魔防 45、速度 105，总和 471。

### 样例定位判断

这只形态的核心定位先暂定为：**物攻先手 / 低费技能节奏输出**。

判断依据：

- 物攻 95 远高于魔攻 35，攻击取向明显偏物攻。
- 速度 105 是非生命五维中的最高项，适合把速度视为核心价值，而不是可牺牲项。
- 特性「挺起胸脯」强化能耗为 1 的技能威力，结合 `energyCycle` 与 `spdLean`，更像依赖出手节奏的低费输出。
- 物防 55、魔防 45 都不高；其中魔防更低，但 `defense` 标签说明耐久并非完全无关，所以弱化防御要保留风险提示。
- 魔攻 35 与形态物攻倾向、特性标签都不匹配，是最适合被弱化的维度。

### 样例打分规则 v0

这是一版用于讨论的人工规则，不代表最终实现：

#### 强化项基础分

| 强化项 | 分数 | 理由 |
|---|---:|---|
| 速度 | +35 | 速度是最高非生命项，并与 `spdLean`、`energyCycle` 同向。 |
| 物攻 | +32 | 物攻是主输出，并与 `patkLean` 同向。 |
| 物防 | +8 | 可补一点物理耐久，但不是核心输出。 |
| 魔防 | +6 | 可补最低防御短板，但收益偏保守。 |
| 魔攻 | -30 | 与物攻形态和标签冲突，强化收益很低。 |

#### 弱化项基础分

| 弱化项 | 分数 | 理由 |
|---|---:|---|
| 魔攻 | +28 | 非核心且数值最低，是最可接受的牺牲项。 |
| 魔防 | +6 | 可以牺牲，但会放大最低防御短板。 |
| 物防 | 0 | 可以牺牲，但会让本就不高的耐久继续变薄。 |
| 物攻 | -35 | 牺牲主输出，通常不接受。 |
| 速度 | -40 | 牺牲先手与低费节奏核心，通常不接受。 |

#### 组合修正

- 速度强化 + 魔攻弱化：额外 +8，最贴合「先手节奏 + 舍弃魔攻」。
- 物攻强化 + 魔攻弱化：额外 +6，最贴合「物攻输出 + 舍弃魔攻」。
- 防御强化 + 魔攻弱化：额外 +2，属于保守可用方案。
- 任意弱化魔防：额外 -2，因为魔防已经偏低。
- 任意弱化物防：额外 -4，因为 `defense` 标签说明耐久仍有价值。

#### 分档规则

- 推荐：总分 ≥ 45，且不弱化物攻 / 速度，不强化魔攻。
- 可保留：总分 ≥ 18，且没有命中硬性风险。
- 不推荐：弱化物攻、弱化速度、强化魔攻，或总分 < 18。
- 不设置单独“均衡”候选；若预置性格不存在同项增减，则只评价 30 个合法性格。

### 这版样例规则暴露出的待讨论点

1. **速度强化 vs 物攻强化谁该首推**：当前规则让速度强化略高于物攻强化，因为特性同时有 `spdLean` 与 `energyCycle`。如果你认为“能耗为 1 的技能威力 +50%”更应优先输出伤害，可以把物攻强化调到第一。
2. **防御标签权重是否过高**：当前 `defense` 只阻止无脑弱化双防，没有把防御强化推到推荐。如果你希望这类精灵更耐久，可以提高物防 / 魔防强化分。
3. **魔防是否可以放心弱化**：魔防是低项，但也可能是可牺牲项。当前规则把弱化魔防列为可保留但有风险；后续可讨论是否应直接降为不推荐。
4. **无均衡性格后的保守方案**：预置性格不允许同项增减，因此后续需要讨论“保守泛用”应由生命/防御强化承担，还是只通过可保留分档表达。

## 待讨论问题

1. “推荐 / 可保留 / 不推荐”更希望按阈值动态分布，还是固定数量？
2. 不存在均衡性格时，“保守泛用”应由生命强化、防御强化还是低风险减益组合来表达？
